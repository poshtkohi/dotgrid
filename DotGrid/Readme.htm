This readme guide you to use DotGrid.DotDFS.

 

DotDFS is a flexible protocol that provides a substantial infrastructure for resource sharing and high-throughput file transfer on current Grid sub-structures and on our DotGrid platform. Our system can be run in both Windows and Linux machines. You can use from our platform in two ways, one programmatically with our extensible APIs (for more information in this regard, see the samples and DotGrid Framework SDK Documentation v1) and another using dotgrid-url-copy Client Utility. The dotgrid-url-copy is a client utility similar to globus-url-copy utility available in Globus Toolkit 4.  For testing DotDFS, first you must run DotGrid.DotDFS servers on your favorite machines that have Microsoft .NET 1.1 for Windows platforms (or newer versions) or MONO .NET 1.1 for Linux platforms (or newer versions) installed. After running the servers, you can use dotgrid-url-copy client utility for running DotDFS client program. The dotgrid-url-copy introduces very commands which can help you for transferring your large files via using parallel TCP connections with high-throughput and high-performance directory tree transfers recursively. You can tell to dotgrid-url-copy that all TCP channels must be forced to be secure based on DotGrid.DotSec protocol similar to HTTPS protocol over SSL3 or TLS1 for your browser in when secure web surfing.

 

Command syntax

The basic syntax dotgrid-url-copy is: 

dotgrid-url-copy -hostname remoteHost -[upload|download] localPath,remotePath -[optional command line switches]

where :

[optional command line switches]

See Command line options below for a list of available options.

remoteHost

Specifies IP or domain name of the destination

DotDFS server for transferring files from or to it.

-upload

Uploads the specified source file or directory tree to

 destination DotDFS server.

-download

Downloads the specified remote source file or directory

 tree from destination DotDFS server.

localPath

Specifies local file name or directory name.

remotePath

Specifies remote file name or directory name.

Note

Any path specifying a directory must end with \ in Windows and / in Linux and Unix.

Command line options

-help

Prints help.

-p <parallelism>

Specifies number of TCP parallel connections for increasing

and aggregating used TCP bandwidth in networks. This option

 increase performance in both in when large file transfers and

 directory tree transfers, significantly.

-secure     

With this parameter all data connections will be forced to

be secure based on DotSec security layer protocol

(Default is non-secure).

-tbs <size>

Specifies the size (in bytes) of the TCP buffer to be usedby

the underlying DotDFS TCP transfer channels. This is critical

to good performance over the WAN.
 

 

How do I choose a value for the TCP buffer size (-tbs) option?

The value you should pick for the TCP buffer size (-tbs) depends on how fast you want to go (your bandwidth) and how far you are moving the data (as measured by the Round Trip Time (RTT) or the time it takes a packet to get to the destination and back.

To calculate the value for -tbs, use the following formula (this assumes that Mega means 1000^2 rather than 1024^2, which is typical for bandwidth):

-tbs = bandwidth in Megabits per second (Mbs) * RTT in milliseconds (ms) * 1000 / 8

As an example, if you are using fast ethernet (100 Mbs) and the RTT was 50 ms it would be:

-tbs = 100 * 50 * 1000 / 8 = 625,000 bytes.

So, how do you come up with values for bandwidth and RTT? To determine RTT, use either ping or traceroute. They both list RTT values.

 

Note

You must be on one end of the transfer and ping the other end. This means that if you are doing a third party transfer you have to run the ping or traceroute between the two server hosts, not from your client.

The bandwidth is a little trickier. Any point in the network can be the bottleneck, so you either need to talk with your network engineers to find out what the bottleneck link is or just assume that your host is the bottleneck and use the speed of your network interface card (NIC).

Note

The value you pick for -tbs limits the top speed you can achieve. You will NOT get bandwidth any higher than what you used in the calculation (assuming the RTT is actually what you specified; it varies a little with network conditions). So, if for some reason you want to limit the bandwidth you get, you can do that by judicious choice of -tbs values.

So where does this formula come from? Because it uses the bandwidth and the RTT (also known as the latency or delay) it is called the bandwidth delay product. The very simple explanation is this: TCP is a reliable protocol. It must save a copy of everything it sends out over the network until the other end acknowledges that it has been received.

As a simple example, if I can put one byte per second onto the network, and it takes 10 seconds for that byte to get there, and 10 seconds for the acknowledgment to get back (RTT = 20 seconds), then I would need at least 20 bytes of storage. Then, hopefully, by the time I am ready to send byte 21, I have received an acknowledgement for byte 1 and I can free that space in my buffer. If you want a more detailed explanation, try the following links on TCP tuning:

  • http://www.psc.edu/networking/perf_tune.html
  • http://www-didc.lbl.gov/TCP-tuning/
  • http://www.ncne.nlanr.net/research/tcp/

How do I choose a value for the parallelism (-p) option?

For most instances, using 4 streams is a very good rule of thumb. Unfortunately, there is not a good formula for picking an exact answer. The shape of the graph shown here is very characteristic in LAN tests:

You get a strong, nearly linear, increase in bandwidth, then a sharp knee, after which additional streams have very little impact. Higher bandwidth, longer round trip times, and more congestion in the network (which you usually can only guess at based on how applications are behaving) will move the knee higher (more streams needed).
